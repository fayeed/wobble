Features

Everything you need to
trust your prompts

Wobble is built for engineers who iterate fast on prompts and need to know — immediately — when something breaks.

📊
Flakiness-corrected pass rates
Run each test case N times. A check only passes if it passes ≥ threshold% of runs. Catches non-deterministic regressions that single-shot testing misses entirely.
runs: 10 · threshold: 0.8
📈
Baseline regression detection
Lock current pass rates as a baseline. Future runs highlight regressions in red and improvements in green. Uses actual pass rates, not just pass/fail booleans.
wobble baseline approve
⚖️
Side-by-side model comparison
Run the full test suite against multiple models concurrently. Get a table of pass rates and costs per model. The killer demo feature.
wobble compare --targets gpt-4o,claude-sonnet-4-6
🕵️
LLM-as-judge evaluation
Use a model to evaluate model output. Supports simple pass/fail criteria, weighted rubric scoring across multiple dimensions, and few-shot calibration examples.
type: llm_judge · rubric mode
📜
Run history & trend tracking
Every run is appended to .wobble/history.jsonl. Drill into any check and see its pass rate over time. First-regression detection tells you exactly when it broke.
wobble history show --test --input --eval
🔠
Variable interpolation
Use {"{{variable}}"} placeholders in prompt files and inputs. Define values per-case. Turns wobble from a toy into something you use on real production prompts.
variables: key: value
👁
Watch mode
Re-run tests automatically on every save to your prompt file or config. 300ms debounce, screen clear, baseline comparison on every run.
wobble watch
🔌
Custom providers & evaluators
Point to any JS module as a provider or evaluator. Wrap internal APIs, fine-tuned models, or edge-deployed inference — no fork required.
provider: ./my-provider.js
🛡
Cost guardrails
Set a max cost per run. Wobble estimates token usage before touching any API and aborts with a clear error if the projected cost exceeds the limit.
limits.max_cost_per_run: 1.00
🔄
Multi-turn conversations
Test full back-and-forth dialogues using the turns: key. Feed history into the model exactly as it would appear in production.
turns: [user, assistant, user]
🏷
Tags & filters
Tag tests and run subsets with --tag smoke or --test my-test. Keep fast smoke tests separate from expensive nightly suites.
tags: [smoke, nightly]
📤
CI-ready output formats
Terminal, JSON, and JUnit XML outputs. Non-zero exit on failure. Plug directly into GitHub Actions, CircleCI, or any CI system that reads JUnit.
--output junit | json | terminal
Evaluators

Eight ways to assert
model output

From simple string matching to weighted rubric scoring — every evaluator composes with N-run flakiness correction.

contains · not_contains
Assert the output contains (or doesn't contain) a substring. Optional case-insensitive matching.
type: contains value: "error" case_sensitive: false
starts_with · ends_with
Assert the output starts or ends with an exact prefix or suffix. Supports case-insensitive mode.
type: starts_with value: "Sure, here" case_sensitive: false
max_length
Assert the output is within a character or word limit. Catches verbose regressions automatically.
type: max_length value: 100 unit: words
regex
Assert the output matches a regular expression. Useful for format validation like dates, IDs, or codes.
type: regex value: '^\d{4}-\d{2}-\d{2}$'
json_schema
Assert the output is valid JSON that conforms to a JSON Schema. Ideal for structured extraction prompts.
type: json_schema schema: type: object required: [name, score]
llm_judge — pass/fail
Use a model to evaluate output against a natural-language criterion. Supports few-shot calibration examples.
type: llm_judge criteria: "Is the reply empathetic?" model: gpt-4o-mini
llm_judge — rubric
Score multiple dimensions 1–10 with configurable weights. Pass if the weighted average meets the threshold.
type: llm_judge threshold: 7 rubric: - dimension: tone weight: 2 - dimension: accuracy weight: 3
custom
Point to any JS module that exports an evaluate function. Full access to the model output and options.
type: custom evaluator: ./my-check.js

Commands

The full
CLI surface

wobble init
Interactive Q&A wizard. Asks about provider, model, flakiness settings, and LLM judge. Generates wobble.yaml and a starter prompt file.
stable
wobble run
Run all tests (or filtered by --test / --tag). Outputs terminal, JSON, or JUnit XML. Compares against baseline if one exists.
stable
wobble watch
Re-run tests on every save to wobble.yaml or any prompt file. 300ms debounce. Shows baseline comparison on each run.
stable
wobble compare
Run the full suite against multiple provider:model targets concurrently. Prints an aligned pass-rate and cost comparison table.
stable
wobble baseline approve
Run tests and lock current pass rates as the new baseline. Always exits 0 — the intent is to record current state, not gate on it.
stable
wobble baseline show
Print stored baseline pass rates grouped by test. --json for a raw dump.
stable
wobble history
Overall pass-rate trend table across all recorded runs. Supports --last N and --json.
stable
wobble history show
Per-check trend with mini bar charts and first-regression detection. Pass --test, --input, --eval to drill in.
stable
Roadmap

What's
coming next

Wobble is in open beta. These features are actively planned — in rough priority order.

🌐
Web UI
Browser dashboard for run history, trend charts, and baseline diffs. No server required — reads your local .wobble/ directory.
📦
Dataset support
Load test cases from CSV or JSONL files. Run wobble against hundreds of real production inputs without hand-writing YAML.
🔗
Prompt chaining
Wire the output of one prompt into the input of another. Test multi-step pipelines end-to-end, not just individual prompts.
⚡
Live watch UI
Full-screen terminal UI for wobble watch — live spinners per test, real-time pass/fail rows, no more scrolling through re-rendered output.
🧩
VS Code extension
Run wobble tests inline from your editor. See pass/fail indicators next to each case in your YAML, with one-click re-run.
🔑
Secret scanning
Warn before committing API keys or PII found in prompt files or test inputs. Integrates with pre-commit hooks automatically.
📊
Cost analytics
Historical cost breakdown by test, model, and provider. Spot which prompts are expensive and whether quality justifies the cost.
🤖
Auto-generated cases
Given a prompt file, generate diverse adversarial test cases automatically using a model. Seed your test suite without writing every case by hand.
Install

Up and running
in 60 seconds

Requires Node ≥ 18. No account, no cloud, no telemetry.

npm
$ npm install -g wobble-cli
pnpm
$ pnpm add -g wobble-cli
# 1. Create your config
$ wobble init

# 2. Set your API key
$ export OPENAI_API_KEY=sk-...

# 3. Run your first test
$ wobble run

# 4. Lock results as baseline
$ wobble baseline approve

# 5. Edit your prompt, catch regressions
$ wobble run
